# RXTrack — Full App Audit & Rework Map

**Date:** 2026-06-19
**Purpose:** Complete analysis of the app's structure, visuals, performance, and code, with a prioritized rework roadmap. This is the map that directs the subsequent sub-project specs.
**Scope note:** The rework is ~5 sub-projects (below). This doc is the audit + sequencing; each sub-project gets its own spec → plan → implementation cycle.

---

## 1. Current structure (as built)

**Shell layout:** top header (~52px) + left **sidebar** + scrollable `<main>`.

- **Sidebar = primary navigation:** a term → block **tree** (collapsible terms, blocks with status icon, lecture count, coverage %). Supports drag-reorder, double-click rename, add term/block, footer stats (questions answered / sessions / lectures). This is a strong, appropriate spine for the domain.
- **Main views** (switched by a single `view` state): `overview`, `block` (default; tabs **objectives** / **lectures**), `study` (the quiz/session engine), `config` (SessionConfig — pre-session setup), `deeplearn` (DeepLearn engine), `tracker` (schedule/coach/weak-areas), `learn`.
- **Orphaned tools** (live only in the top-right **profile menu** modals, disconnected from study): 🩺 **Patient Recognition**, 🃏 **Anki Sync**, 📋 **Study Routine**.

**The learning spine works:** term → block → lecture → configure → study/quiz → tracker. The problem is not the spine; it's that newer pedagogy tools hang off a profile menu instead of living inside that spine.

---

## 2. Findings by area

### 2.1 Information architecture / navigation
- **Strength:** sidebar term/block tree is the right mental model; coverage % inline is good feedback.
- **Issue — orphaned tools:** Patient Recognition / Anki Sync / Study Routine are buried in the profile menu. Recognition especially is *reference + active recall during study* — it belongs in the block/lecture/study context, not a settings dropdown.
- **Issue — flat view switch:** `view` is one string toggling a giant conditional. `overview` vs `block` vs `learn` overlap in purpose and aren't clearly differentiated to the user. Some views (`learn`, `overview`) appear under-used vs `block`/`study`/`tracker`.
- **Issue — no contextual surfacing:** while on a block/lecture, there's no consistent "study this / quiz this / recognize this / routine" action set; entry points are scattered.

### 2.2 Visual system
- **Tokens exist** (`theme.js`: colorblind-safe status palette, dark/light) and fonts landed (IBM Plex Sans/Mono + Fraunces, `--font-*` vars). Good foundation.
- **Issue — inline styles everywhere:** essentially every element uses an ad-hoc `style={{…}}` object. There are no reusable styled primitives (Button, Card, Panel, Field, Badge). Consequences: (a) visual inconsistency (spacing/radius/shadow vary per element), (b) any restyle = thousands of edits, (c) `App.jsx` bloat, (d) hard to enforce the token system.
- **Issue — density/rhythm:** spacing, font sizes, radii are chosen per-element (10px here, 12px there; radius 5/7/9/10/11/16 all appear). No spacing scale, no type scale applied consistently.
- **Opportunity:** a small component kit + spacing/type scales would unify the look *and* shrink the code.

### 2.3 Performance
- **Single ~2.66 MB JS bundle, zero code-splitting** (no `React.lazy`/`Suspense`). Everything — Tracker, DeepLearn, Session engine, Patient Recognition, OCR/Mistral — loads before first paint.
- **Biggest, lowest-risk win:** route-level code-splitting per view (`lazy(() => import('./Tracker'))`, etc.) + `manualChunks`. Should cut initial load substantially.
- **Secondary:** heavy prop-drilling forces wide re-renders; memoization is present (121 `useMemo`/`useCallback` in App.jsx) but reactive boundaries are unclear because state lives in one component.

### 2.4 Code architecture
- **`App.jsx` is 40,292 lines** — a monolith holding global state, dozens of helpers, the sidebar, every view's render, and most modals. (`Tracker.jsx` 10.5k, `DeepLearn.jsx` 9.2k are also large.)
- **Heavy prop-drilling:** e.g. `Tracker` receives ~35 props; `Session`, `SessionConfig` similar. No context/store — adding/threading any cross-cutting value touches many call sites.
- **Consequence:** slow to evolve, risky to edit, hard for tooling/builds. This is the root cause behind both the "improved speed" and "hard to rework visuals" pains.

### 2.5 Feature surface (keep / merge / remove — to confirm with user)
- **Keep (core):** sidebar tree, block view, Session/quiz engine, Tracker, DeepLearn, objectives, Anki ingest + Recognition bank.
- **Examine for overlap:** `overview` vs `block` vs `learn` — likely consolidate to one clear "block home."
- **Reposition:** Patient Recognition, Study Routine, Anki Sync out of profile menu into the flow (Anki Sync is a setup action → settings is OK; Recognition + Routine → study context).
- **Confirm usage:** any half-built views the user no longer uses (candidate removals).

## 3. Rework roadmap (sequenced sub-projects)

Each is its own spec → plan → build. Ordered for least-rework and fastest daily payoff.

1. **Performance / code-split** *(parallel, low-risk, do early)* — route-level `lazy`+`Suspense` per view, `manualChunks`, lazy-load heavy/rare modules (OCR, Recognition). No visual change; immediate speed.
2. **Component kit + design tokens applied** — build Button/Card/Panel/Field/Badge/Modal primitives bound to the token system + spacing/type scales. Foundation for all visual work; incrementally replaces inline styles. Shrinks `App.jsx` as a side effect.
3. **IA / navigation + recognition placement** — consolidate overlapping views into a clear block home; define a consistent per-context action set (Study / Quiz / Recognize / Routine); move orphaned tools into the flow. *Depends on #2 for the new surfaces.*
4. **Visual overhaul** — apply the kit broadly: refined typography rhythm, spacing scale, color/elevation, dark mode polish, empty/loading states. *Builds on #2 + #3.*
5. **Recognition generation strategy** *(separate, already half-built)* — fix the Edge Function batching/timeout, paging, and the scale/cost strategy (which cards, weak-area-driven) for the bank. Folds into how Recognition surfaces in #3.

**Cross-cutting (incremental, every sub-project):** carve `App.jsx` into focused modules as each area is touched — extract the sidebar, each view, and shared state — rather than one big-bang refactor.

---

## 4. Recommended starting order

- **Track A (now):** #1 Performance — isolated, safe, instant payoff, unblocks nothing else.
- **Track B (now):** #2 Component kit — the foundation everything visual depends on.
- Then #3 IA/navigation (with recognition placement), then #4 visual overhaul, with #5 recognition-generation slotted when Recognition surfaces.

Open decisions for the user (Section 2.5 + sequencing) are resolved in each sub-project's own brainstorm; this doc only sets the map and order.
